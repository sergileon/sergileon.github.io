39  /**
40   * <p>A {@link Query} that matches numeric values within a
41   * specified range.  To use this, you must first index the
42   * numeric values using {@link IntField}, {@link
43   * FloatField}, {@link LongField} or {@link DoubleField} (expert: {@link
44   * NumericTokenStream}).  If your terms are instead textual,
45   * you should use {@link TermRangeQuery}.</p>
46   *
47   * <p>You create a new NumericRangeQuery with the static
48   * factory methods, eg:
49   *
50   * <pre class="prettyprint">
51   * Query q = NumericRangeQuery.newFloatRange("weight", 0.03f, 0.10f, true, true);
52   * </pre>
53   *
54   * matches all documents whose float valued "weight" field
55   * ranges from 0.03 to 0.10, inclusive.
56   *
57   * <p>The performance of NumericRangeQuery is much better
58   * than the corresponding {@link TermRangeQuery} because the
59   * number of terms that must be searched is usually far
60   * fewer, thanks to trie indexing, described below.</p>
61   *
62   * <p>You can optionally specify a <a
63   * href="#precisionStepDesc"><code>precisionStep</code></a>
64   * when creating this query.  This is necessary if you've
65   * changed this configuration from its default (4) during
66   * indexing.  Lower values consume more disk space but speed
67   * up searching.  Suitable values are between <b>1</b> and
68   * <b>8</b>. A good starting point to test is <b>4</b>,
69   * which is the default value for all <code>Numeric*</code>
70   * classes.  See <a href="#precisionStepDesc">below</a> for
71   * details.
72   *
73   * <p>This query defaults to {@linkplain
74   * MultiTermQuery#CONSTANT_SCORE_REWRITE}.
75   * With precision steps of &le;4, this query can be run with
76   * one of the BooleanQuery rewrite methods without changing
77   * BooleanQuery's default max clause count.
78   *
79   * <br><h3>How it works</h3>
80   *
81   * <p>See the publication about <a target="_blank" href="http://www.panfmp.org">panFMP</a>,
82   * where this algorithm was described (referred to as <code>TrieRangeQuery</code>):
83   *
84   * <blockquote><strong>Schindler, U, Diepenbroek, M</strong>, 2008.
85   * <em>Generic XML-based Framework for Metadata Portals.</em>
86   * Computers &amp; Geosciences 34 (12), 1947-1955.
87   * <a href="http://dx.doi.org/10.1016/j.cageo.2008.02.023"
88   * target="_blank">doi:10.1016/j.cageo.2008.02.023</a></blockquote>
89   *
90   * <p><em>A quote from this paper:</em> Because Apache Lucene is a full-text
91   * search engine and not a conventional database, it cannot handle numerical ranges
92   * (e.g., field value is inside user defined bounds, even dates are numerical values).
93   * We have developed an extension to Apache Lucene that stores
94   * the numerical values in a special string-encoded format with variable precision
95   * (all numerical values like doubles, longs, floats, and ints are converted to
96   * lexicographic sortable string representations and stored with different precisions
97   * (for a more detailed description of how the values are stored,
98   * see {@link NumericUtils}). A range is then divided recursively into multiple intervals for searching:
99   * The center of the range is searched only with the lowest possible precision in the <em>trie</em>,
100  * while the boundaries are matched more exactly. This reduces the number of terms dramatically.</p>
101  *
102  * <p>For the variant that stores long values in 8 different precisions (each reduced by 8 bits) that
103  * uses a lowest precision of 1 byte, the index contains only a maximum of 256 distinct values in the
104  * lowest precision. Overall, a range could consist of a theoretical maximum of
105  * <code>7*255*2 + 255 = 3825</code> distinct terms (when there is a term for every distinct value of an
106  * 8-byte-number in the index and the range covers almost all of them; a maximum of 255 distinct values is used
107  * because it would always be possible to reduce the full 256 values to one term with degraded precision).
108  * In practice, we have seen up to 300 terms in most cases (index with 500,000 metadata records
109  * and a uniform value distribution).</p>
110  *
111  * <h3><a name="precisionStepDesc">Precision Step</a></h3>
112  * <p>You can choose any <code>precisionStep</code> when encoding values.
113  * Lower step values mean more precisions and so more terms in index (and index gets larger). The number
114  * of indexed terms per value is (those are generated by {@link NumericTokenStream}):
115  * <p style="font-family:serif">
116  * &nbsp;&nbsp;indexedTermsPerValue = <b>ceil</b><big>(</big>bitsPerValue / precisionStep<big>)</big>
117  * </p>
118  * As the lower precision terms are shared by many values, the additional terms only
119  * slightly grow the term dictionary (approx. 7% for <code>precisionStep=4</code>), but have a larger
120  * impact on the postings (the postings file will have  more entries, as every document is linked to
121  * <code>indexedTermsPerValue</code> terms instead of one). The formula to estimate the growth
122  * of the term dictionary in comparison to one term per value:
123  * <p>
124  * <!-- the formula in the alt attribute was transformed from latex to PNG with http://1.618034.com/latex.php (with 110 dpi): -->
125  * &nbsp;&nbsp;<img src="doc-files/nrq-formula-1.png" alt="\mathrm{termDictOverhead} = \sum\limits_{i=0}^{\mathrm{indexedTermsPerValue}-1} \frac{1}{2^{\mathrm{precisionStep}\cdot i}}">
126  * </p>
127  * <p>On the other hand, if the <code>precisionStep</code> is smaller, the maximum number of terms to match reduces,
128  * which optimizes query speed. The formula to calculate the maximum number of terms that will be visited while
129  * executing the query is:
130  * <p>
131  * <!-- the formula in the alt attribute was transformed from latex to PNG with http://1.618034.com/latex.php (with 110 dpi): -->
132  * &nbsp;&nbsp;<img src="doc-files/nrq-formula-2.png" alt="\mathrm{maxQueryTerms} = \left[ \left( \mathrm{indexedTermsPerValue} - 1 \right) \cdot \left(2^\mathrm{precisionStep} - 1 \right) \cdot 2 \right] + \left( 2^\mathrm{precisionStep} - 1 \right)">
133  * </p>
134  * <p>For longs stored using a precision step of 4, <code>maxQueryTerms = 15*15*2 + 15 = 465</code>, and for a precision
135  * step of 2, <code>maxQueryTerms = 31*3*2 + 3 = 189</code>. But the faster search speed is reduced by more seeking
136  * in the term enum of the index. Because of this, the ideal <code>precisionStep</code> value can only
137  * be found out by testing. <b>Important:</b> You can index with a lower precision step value and test search speed
138  * using a multiple of the original step value.</p>
139  *
140  * <p>Good values for <code>precisionStep</code> are depending on usage and data type:
141  * <ul>
142  *  <li>The default for all data types is <b>4</b>, which is used, when no <code>precisionStep</code> is given.
143  *  <li>Ideal value in most cases for <em>64 bit</em> data types <em>(long, double)</em> is <b>6</b> or <b>8</b>.
144  *  <li>Ideal value in most cases for <em>32 bit</em> data types <em>(int, float)</em> is <b>4</b>.
145  *  <li>For low cardinality fields larger precision steps are good. If the cardinality is &lt; 100, it is
146  *  fair to use {@link Integer#MAX_VALUE} (see below).
147  *  <li>Steps <b>&ge;64</b> for <em>long/double</em> and <b>&ge;32</b> for <em>int/float</em> produces one token
148  *  per value in the index and querying is as slow as a conventional {@link TermRangeQuery}. But it can be used
149  *  to produce fields, that are solely used for sorting (in this case simply use {@link Integer#MAX_VALUE} as
150  *  <code>precisionStep</code>). Using {@link IntField},
151  *  {@link LongField}, {@link FloatField} or {@link DoubleField} for sorting
152  *  is ideal, because building the field cache is much faster than with text-only numbers.
153  *  These fields have one term per value and therefore also work with term enumeration for building distinct lists
154  *  (e.g. facets / preselected values to search for).
155  *  Sorting is also possible with range query optimized fields using one of the above <code>precisionSteps</code>.
156  * </ul>
157  *
158  * <p>Comparisons of the different types of RangeQueries on an index with about 500,000 docs showed
159  * that {@link TermRangeQuery} in boolean rewrite mode (with raised {@link BooleanQuery} clause count)
160  * took about 30-40 secs to complete, {@link TermRangeQuery} in constant score filter rewrite mode took 5 secs
161  * and executing this class took &lt;100ms to complete (on an Opteron64 machine, Java 1.5, 8 bit
162  * precision step). This query type was developed for a geographic portal, where the performance for
163  * e.g. bounding boxes or exact date/time stamps is important.</p>
164  *
165  * @since 2.9
166  **/
167 public final class NumericRangeQuery<T extends Number> extends MultiTermQuery {

169   private NumericRangeQuery(final String field, final int precisionStep, final NumericType dataType,
170     T min, T max, final boolean minInclusive, final boolean maxInclusive) {
171     super(field);
172     if (precisionStep < 1)
173       throw new IllegalArgumentException("precisionStep must be >=1");
174     this.precisionStep = precisionStep;
175     this.dataType = Objects.requireNonNull(dataType, "NumericType must not be null");
176     this.min = min;
177     this.max = max;
178     this.minInclusive = minInclusive;
179     this.maxInclusive = maxInclusive;
180   }

377   /**
378    * Subclass of FilteredTermsEnum for enumerating all terms that match the
379    * sub-ranges for trie range queries, using flex API.
380    * <p>
381    * WARNING: This term enumeration is not guaranteed to be always ordered by
382    * {@link Term#compareTo}.
383    * The ordering depends on how {@link NumericUtils#splitLongRange} and
384    * {@link NumericUtils#splitIntRange} generates the sub-ranges. For
385    * {@link MultiTermQuery} ordering is not relevant.
386    */
387   private final class NumericRangeTermsEnum extends FilteredTermsEnum {
388 
389     private BytesRef currentLowerBound, currentUpperBound;
390 
391     private final LinkedList<BytesRef> rangeBounds = new LinkedList<>();
392 
393     NumericRangeTermsEnum(final TermsEnum tenum) {
394       super(tenum);
395       switch (dataType) {
396         case LONG:
397         case DOUBLE: {
398           // lower
399           long minBound;
400           if (dataType == NumericType.LONG) {
401             minBound = (min == null) ? Long.MIN_VALUE : min.longValue();
402           } else {
403             assert dataType == NumericType.DOUBLE;
404             minBound = (min == null) ? LONG_NEGATIVE_INFINITY
405               : NumericUtils.doubleToSortableLong(min.doubleValue());
406           }
407           if (!minInclusive && min != null) {
408             if (minBound == Long.MAX_VALUE) break;
409             minBound++;
410           }
411           
412           // upper
413           long maxBound;
414           if (dataType == NumericType.LONG) {
415             maxBound = (max == null) ? Long.MAX_VALUE : max.longValue();
416           } else {
417             assert dataType == NumericType.DOUBLE;
418             maxBound = (max == null) ? LONG_POSITIVE_INFINITY
419               : NumericUtils.doubleToSortableLong(max.doubleValue());
420           }
421           if (!maxInclusive && max != null) {
422             if (maxBound == Long.MIN_VALUE) break;
423             maxBound--;
424           }
425           
426           NumericUtils.splitLongRange(new NumericUtils.LongRangeBuilder() {
427             @Override
428             public final void addRange(BytesRef minPrefixCoded, BytesRef maxPrefixCoded) {
429               rangeBounds.add(minPrefixCoded);
430               rangeBounds.add(maxPrefixCoded);
431             }
432           }, precisionStep, minBound, maxBound);
433           break;
434         }
435           
436         case INT:
437         case FLOAT: {
438           // lower
439           int minBound;
440           if (dataType == NumericType.INT) {
441             minBound = (min == null) ? Integer.MIN_VALUE : min.intValue();
442           } else {
443             assert dataType == NumericType.FLOAT;
444             minBound = (min == null) ? INT_NEGATIVE_INFINITY
445               : NumericUtils.floatToSortableInt(min.floatValue());
446           }
447           if (!minInclusive && min != null) {
448             if (minBound == Integer.MAX_VALUE) break;
449             minBound++;
450           }
451           
452           // upper
453           int maxBound;
454           if (dataType == NumericType.INT) {
455             maxBound = (max == null) ? Integer.MAX_VALUE : max.intValue();
456           } else {
457             assert dataType == NumericType.FLOAT;
458             maxBound = (max == null) ? INT_POSITIVE_INFINITY
459               : NumericUtils.floatToSortableInt(max.floatValue());
460           }
461           if (!maxInclusive && max != null) {
462             if (maxBound == Integer.MIN_VALUE) break;
463             maxBound--;
464           }
465           
466           NumericUtils.splitIntRange(new NumericUtils.IntRangeBuilder() {
467             @Override
468             public final void addRange(BytesRef minPrefixCoded, BytesRef maxPrefixCoded) {
469               rangeBounds.add(minPrefixCoded);
470               rangeBounds.add(maxPrefixCoded);
471             }
472           }, precisionStep, minBound, maxBound);
473           break;
474         }
475           
476         default:
477           // should never happen
478           throw new IllegalArgumentException("Invalid NumericType");
479       }
480     }
481     
482     private void nextRange() {
483       assert rangeBounds.size() % 2 == 0;
484 
485       currentLowerBound = rangeBounds.removeFirst();
486       assert currentUpperBound == null || currentUpperBound.compareTo(currentLowerBound) <= 0 :
487         "The current upper bound must be <= the new lower bound";
488       
489       currentUpperBound = rangeBounds.removeFirst();
490     }
491     
492     @Override
493     protected final BytesRef nextSeekTerm(BytesRef term) {
494       while (rangeBounds.size() >= 2) {
495         nextRange();
496         
497         // if the new upper bound is before the term parameter, the sub-range is never a hit
498         if (term != null && term.compareTo(currentUpperBound) > 0)
499           continue;
500         // never seek backwards, so use current term if lower bound is smaller
501         return (term != null && term.compareTo(currentLowerBound) > 0) ?
502           term : currentLowerBound;
503       }
504       
505       // no more sub-range enums available
506       assert rangeBounds.isEmpty();
507       currentLowerBound = currentUpperBound = null;
508       return null;
509     }
510     
511     @Override
512     protected final AcceptStatus accept(BytesRef term) {
513       while (currentUpperBound == null || term.compareTo(currentUpperBound) > 0) {
514         if (rangeBounds.isEmpty())
515           return AcceptStatus.END;
516         // peek next sub-range, only seek if the current term is smaller than next lower bound
517         if (term.compareTo(rangeBounds.getFirst()) < 0)
518           return AcceptStatus.NO_AND_SEEK;
519         // step forward to next range without seeking, as next lower range bound is less or equal current term
520         nextRange();
521       }
522       return AcceptStatus.YES;
523     }
