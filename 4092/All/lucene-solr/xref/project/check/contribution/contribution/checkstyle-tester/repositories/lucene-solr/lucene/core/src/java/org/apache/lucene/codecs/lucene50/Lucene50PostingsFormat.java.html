42  /**
43   * Lucene 5.0 postings format, which encodes postings in packed integer blocks 
44   * for fast decode.
45   *
46   * <p>
47   * Basic idea:
48   * <ul>
49   *   <li>
50   *   <b>Packed Blocks and VInt Blocks</b>: 
51   *   <p>In packed blocks, integers are encoded with the same bit width ({@link PackedInts packed format}):
52   *      the block size (i.e. number of integers inside block) is fixed (currently 128). Additionally blocks
53   *      that are all the same value are encoded in an optimized way.</p>
54   *   <p>In VInt blocks, integers are encoded as {@link DataOutput#writeVInt VInt}:
55   *      the block size is variable.</p>
56   *   </li>
57   *
58   *   <li> 
59   *   <b>Block structure</b>: 
60   *   <p>When the postings are long enough, Lucene50PostingsFormat will try to encode most integer data 
61   *      as a packed block.</p> 
62   *   <p>Take a term with 259 documents as an example, the first 256 document ids are encoded as two packed 
63   *      blocks, while the remaining 3 are encoded as one VInt block. </p>
64   *   <p>Different kinds of data are always encoded separately into different packed blocks, but may 
65   *      possibly be interleaved into the same VInt block. </p>
66   *   <p>This strategy is applied to pairs: 
67   *      &lt;document number, frequency&gt;,
68   *      &lt;position, payload length&gt;, 
69   *      &lt;position, offset start, offset length&gt;, and
70   *      &lt;position, payload length, offsetstart, offset length&gt;.</p>
71   *   </li>
72   *
73   *   <li>
74   *   <b>Skipdata settings</b>: 
75   *   <p>The structure of skip table is quite similar to previous version of Lucene. Skip interval is the 
76   *      same as block size, and each skip entry points to the beginning of each block. However, for 
77   *      the first block, skip data is omitted.</p>
78   *   </li>
79   *
80   *   <li>
81   *   <b>Positions, Payloads, and Offsets</b>: 
82   *   <p>A position is an integer indicating where the term occurs within one document. 
83   *      A payload is a blob of metadata associated with current position. 
84   *      An offset is a pair of integers indicating the tokenized start/end offsets for given term 
85   *      in current position: it is essentially a specialized payload. </p>
86   *   <p>When payloads and offsets are not omitted, numPositions==numPayloads==numOffsets (assuming a 
87   *      null payload contributes one count). As mentioned in block structure, it is possible to encode 
88   *      these three either combined or separately. 
89   *   <p>In all cases, payloads and offsets are stored together. When encoded as a packed block, 
90   *      position data is separated out as .pos, while payloads and offsets are encoded in .pay (payload 
91   *      metadata will also be stored directly in .pay). When encoded as VInt blocks, all these three are 
92   *      stored interleaved into the .pos (so is payload metadata).</p>
93   *   <p>With this strategy, the majority of payload and offset data will be outside .pos file. 
94   *      So for queries that require only position data, running on a full index with payloads and offsets, 
95   *      this reduces disk pre-fetches.</p>
96   *   </li>
97   * </ul>
98   *
99   * <p>
100  * Files and detailed format:
101  * <ul>
102  *   <li><tt>.tim</tt>: <a href="#Termdictionary">Term Dictionary</a></li>
103  *   <li><tt>.tip</tt>: <a href="#Termindex">Term Index</a></li>
104  *   <li><tt>.doc</tt>: <a href="#Frequencies">Frequencies and Skip Data</a></li>
105  *   <li><tt>.pos</tt>: <a href="#Positions">Positions</a></li>
106  *   <li><tt>.pay</tt>: <a href="#Payloads">Payloads and Offsets</a></li>
107  * </ul>
108  *
109  * <a name="Termdictionary"></a>
110  * <dl>
111  * <dd>
112  * <b>Term Dictionary</b>
113  *
114  * <p>The .tim file contains the list of terms in each
115  * field along with per-term statistics (such as docfreq)
116  * and pointers to the frequencies, positions, payload and
117  * skip data in the .doc, .pos, and .pay files.
118  * See {@link BlockTreeTermsWriter} for more details on the format.
119  *
120  * <p>NOTE: The term dictionary can plug into different postings implementations:
121  * the postings writer/reader are actually responsible for encoding 
122  * and decoding the PostingsHeader and TermMetadata sections described here:
123  *
124  * <ul>
125  *   <li>PostingsHeader --&gt; Header, PackedBlockSize</li>
126  *   <li>TermMetadata --&gt; (DocFPDelta|SingletonDocID), PosFPDelta?, PosVIntBlockFPDelta?, PayFPDelta?, 
127  *                            SkipFPDelta?</li>
128  *   <li>Header, --&gt; {@link CodecUtil#writeIndexHeader IndexHeader}</li>
129  *   <li>PackedBlockSize, SingletonDocID --&gt; {@link DataOutput#writeVInt VInt}</li>
130  *   <li>DocFPDelta, PosFPDelta, PayFPDelta, PosVIntBlockFPDelta, SkipFPDelta --&gt; {@link DataOutput#writeVLong VLong}</li>
131  *   <li>Footer --&gt; {@link CodecUtil#writeFooter CodecFooter}</li>
132  * </ul>
133  * <p>Notes:
134  * <ul>
135  *    <li>Header is a {@link CodecUtil#writeIndexHeader IndexHeader} storing the version information
136  *        for the postings.</li>
137  *    <li>PackedBlockSize is the fixed block size for packed blocks. In packed block, bit width is 
138  *        determined by the largest integer. Smaller block size result in smaller variance among width 
139  *        of integers hence smaller indexes. Larger block size result in more efficient bulk i/o hence
140  *        better acceleration. This value should always be a multiple of 64, currently fixed as 128 as 
141  *        a tradeoff. It is also the skip interval used to accelerate {@link org.apache.lucene.index.PostingsEnum#advance(int)}.
142  *    <li>DocFPDelta determines the position of this term's TermFreqs within the .doc file. 
143  *        In particular, it is the difference of file offset between this term's
144  *        data and previous term's data (or zero, for the first term in the block).On disk it is 
145  *        stored as the difference from previous value in sequence. </li>
146  *    <li>PosFPDelta determines the position of this term's TermPositions within the .pos file.
147  *        While PayFPDelta determines the position of this term's &lt;TermPayloads, TermOffsets?&gt; within 
148  *        the .pay file. Similar to DocFPDelta, it is the difference between two file positions (or 
149  *        neglected, for fields that omit payloads and offsets).</li>
150  *    <li>PosVIntBlockFPDelta determines the position of this term's last TermPosition in last pos packed
151  *        block within the .pos file. It is synonym for PayVIntBlockFPDelta or OffsetVIntBlockFPDelta. 
152  *        This is actually used to indicate whether it is necessary to load following
153  *        payloads and offsets from .pos instead of .pay. Every time a new block of positions are to be 
154  *        loaded, the PostingsReader will use this value to check whether current block is packed format
155  *        or VInt. When packed format, payloads and offsets are fetched from .pay, otherwise from .pos. 
156  *        (this value is neglected when total number of positions i.e. totalTermFreq is less or equal 
157  *        to PackedBlockSize).
158  *    <li>SkipFPDelta determines the position of this term's SkipData within the .doc
159  *        file. In particular, it is the length of the TermFreq data.
160  *        SkipDelta is only stored if DocFreq is not smaller than SkipMinimum
161  *        (i.e. 128 in Lucene50PostingsFormat).</li>
162  *    <li>SingletonDocID is an optimization when a term only appears in one document. In this case, instead
163  *        of writing a file pointer to the .doc file (DocFPDelta), and then a VIntBlock at that location, the 
164  *        single document ID is written to the term dictionary.</li>
165  * </ul>
166  * </dd>
167  * </dl>
168  *
169  * <a name="Termindex"></a>
170  * <dl>
171  * <dd>
172  * <b>Term Index</b>
173  * <p>The .tip file contains an index into the term dictionary, so that it can be 
174  * accessed randomly.  See {@link BlockTreeTermsWriter} for more details on the format.
175  * </dd>
176  * </dl>
177  *
178  *
179  * <a name="Frequencies"></a>
180  * <dl>
181  * <dd>
182  * <b>Frequencies and Skip Data</b>
183  *
184  * <p>The .doc file contains the lists of documents which contain each term, along
185  * with the frequency of the term in that document (except when frequencies are
186  * omitted: {@link IndexOptions#DOCS}). It also saves skip data to the beginning of 
187  * each packed or VInt block, when the length of document list is larger than packed block size.</p>
188  *
189  * <ul>
190  *   <li>docFile(.doc) --&gt; Header, &lt;TermFreqs, SkipData?&gt;<sup>TermCount</sup>, Footer</li>
191  *   <li>Header --&gt; {@link CodecUtil#writeIndexHeader IndexHeader}</li>
192  *   <li>TermFreqs --&gt; &lt;PackedBlock&gt; <sup>PackedDocBlockNum</sup>,  
193  *                        VIntBlock? </li>
194  *   <li>PackedBlock --&gt; PackedDocDeltaBlock, PackedFreqBlock?
195  *   <li>VIntBlock --&gt; &lt;DocDelta[, Freq?]&gt;<sup>DocFreq-PackedBlockSize*PackedDocBlockNum</sup>
196  *   <li>SkipData --&gt; &lt;&lt;SkipLevelLength, SkipLevel&gt;
197  *       <sup>NumSkipLevels-1</sup>, SkipLevel&gt;, SkipDatum?</li>
198  *   <li>SkipLevel --&gt; &lt;SkipDatum&gt; <sup>TrimmedDocFreq/(PackedBlockSize^(Level + 1))</sup></li>
199  *   <li>SkipDatum --&gt; DocSkip, DocFPSkip, &lt;PosFPSkip, PosBlockOffset, PayLength?, 
200  *                        PayFPSkip?&gt;?, SkipChildLevelPointer?</li>
201  *   <li>PackedDocDeltaBlock, PackedFreqBlock --&gt; {@link PackedInts PackedInts}</li>
202  *   <li>DocDelta, Freq, DocSkip, DocFPSkip, PosFPSkip, PosBlockOffset, PayByteUpto, PayFPSkip 
203  *       --&gt; 
204  *   {@link DataOutput#writeVInt VInt}</li>
205  *   <li>SkipChildLevelPointer --&gt; {@link DataOutput#writeVLong VLong}</li>
206  *   <li>Footer --&gt; {@link CodecUtil#writeFooter CodecFooter}</li>
207  * </ul>
208  * <p>Notes:
209  * <ul>
210  *   <li>PackedDocDeltaBlock is theoretically generated from two steps: 
211  *     <ol>
212  *       <li>Calculate the difference between each document number and previous one, 
213  *           and get a d-gaps list (for the first document, use absolute value); </li>
214  *       <li>For those d-gaps from first one to PackedDocBlockNum*PackedBlockSize<sup>th</sup>, 
215  *           separately encode as packed blocks.</li>
216  *     </ol>
217  *     If frequencies are not omitted, PackedFreqBlock will be generated without d-gap step.
218  *   </li>
219  *   <li>VIntBlock stores remaining d-gaps (along with frequencies when possible) with a format 
220  *       that encodes DocDelta and Freq:
221  *       <p>DocDelta: if frequencies are indexed, this determines both the document
222  *       number and the frequency. In particular, DocDelta/2 is the difference between
223  *       this document number and the previous document number (or zero when this is the
224  *       first document in a TermFreqs). When DocDelta is odd, the frequency is one.
225  *       When DocDelta is even, the frequency is read as another VInt. If frequencies
226  *       are omitted, DocDelta contains the gap (not multiplied by 2) between document
227  *       numbers and no frequency information is stored.</p>
228  *       <p>For example, the TermFreqs for a term which occurs once in document seven
229  *          and three times in document eleven, with frequencies indexed, would be the
230  *          following sequence of VInts:</p>
231  *       <p>15, 8, 3</p>
232  *       <p>If frequencies were omitted ({@link IndexOptions#DOCS}) it would be this
233  *          sequence of VInts instead:</p>
234  *       <p>7,4</p>
235  *   </li>
236  *   <li>PackedDocBlockNum is the number of packed blocks for current term's docids or frequencies. 
237  *       In particular, PackedDocBlockNum = floor(DocFreq/PackedBlockSize) </li>
238  *   <li>TrimmedDocFreq = DocFreq % PackedBlockSize == 0 ? DocFreq - 1 : DocFreq. 
239  *       We use this trick since the definition of skip entry is a little different from base interface.
240  *       In {@link MultiLevelSkipListWriter}, skip data is assumed to be saved for
241  *       skipInterval<sup>th</sup>, 2*skipInterval<sup>th</sup> ... posting in the list. However, 
242  *       in Lucene50PostingsFormat, the skip data is saved for skipInterval+1<sup>th</sup>, 
243  *       2*skipInterval+1<sup>th</sup> ... posting (skipInterval==PackedBlockSize in this case). 
244  *       When DocFreq is multiple of PackedBlockSize, MultiLevelSkipListWriter will expect one 
245  *       more skip data than Lucene50SkipWriter. </li>
246  *   <li>SkipDatum is the metadata of one skip entry.
247  *      For the first block (no matter packed or VInt), it is omitted.</li>
248  *   <li>DocSkip records the document number of every PackedBlockSize<sup>th</sup> document number in
249  *       the postings (i.e. last document number in each packed block). On disk it is stored as the 
250  *       difference from previous value in the sequence. </li>
251  *   <li>DocFPSkip records the file offsets of each block (excluding )posting at 
252  *       PackedBlockSize+1<sup>th</sup>, 2*PackedBlockSize+1<sup>th</sup> ... , in DocFile. 
253  *       The file offsets are relative to the start of current term's TermFreqs. 
254  *       On disk it is also stored as the difference from previous SkipDatum in the sequence.</li>
255  *   <li>Since positions and payloads are also block encoded, the skip should skip to related block first,
256  *       then fetch the values according to in-block offset. PosFPSkip and PayFPSkip record the file 
257  *       offsets of related block in .pos and .pay, respectively. While PosBlockOffset indicates
258  *       which value to fetch inside the related block (PayBlockOffset is unnecessary since it is always
259  *       equal to PosBlockOffset). Same as DocFPSkip, the file offsets are relative to the start of 
260  *       current term's TermFreqs, and stored as a difference sequence.</li>
261  *   <li>PayByteUpto indicates the start offset of the current payload. It is equivalent to
262  *       the sum of the payload lengths in the current block up to PosBlockOffset</li>
263  * </ul>
264  * </dd>
265  * </dl>
266  *
267  * <a name="Positions"></a>
268  * <dl>
269  * <dd>
270  * <b>Positions</b>
271  * <p>The .pos file contains the lists of positions that each term occurs at within documents. It also
272  *    sometimes stores part of payloads and offsets for speedup.</p>
273  * <ul>
274  *   <li>PosFile(.pos) --&gt; Header, &lt;TermPositions&gt; <sup>TermCount</sup>, Footer</li>
275  *   <li>Header --&gt; {@link CodecUtil#writeIndexHeader IndexHeader}</li>
276  *   <li>TermPositions --&gt; &lt;PackedPosDeltaBlock&gt; <sup>PackedPosBlockNum</sup>,  
277  *                            VIntBlock? </li>
278  *   <li>VIntBlock --&gt; &lt;PositionDelta[, PayloadLength?], PayloadData?, 
279  *                        OffsetDelta?, OffsetLength?&gt;<sup>PosVIntCount</sup>
280  *   <li>PackedPosDeltaBlock --&gt; {@link PackedInts PackedInts}</li>
281  *   <li>PositionDelta, OffsetDelta, OffsetLength --&gt; 
282  *       {@link DataOutput#writeVInt VInt}</li>
283  *   <li>PayloadData --&gt; {@link DataOutput#writeByte byte}<sup>PayLength</sup></li>
284  *   <li>Footer --&gt; {@link CodecUtil#writeFooter CodecFooter}</li>
285  * </ul>
286  * <p>Notes:
287  * <ul>
288  *   <li>TermPositions are order by term (terms are implicit, from the term dictionary), and position 
289  *       values for each term document pair are incremental, and ordered by document number.</li>
290  *   <li>PackedPosBlockNum is the number of packed blocks for current term's positions, payloads or offsets. 
291  *       In particular, PackedPosBlockNum = floor(totalTermFreq/PackedBlockSize) </li>
292  *   <li>PosVIntCount is the number of positions encoded as VInt format. In particular, 
293  *       PosVIntCount = totalTermFreq - PackedPosBlockNum*PackedBlockSize</li>
294  *   <li>The procedure how PackedPosDeltaBlock is generated is the same as PackedDocDeltaBlock 
295  *       in chapter <a href="#Frequencies">Frequencies and Skip Data</a>.</li>
296  *   <li>PositionDelta is, if payloads are disabled for the term's field, the
297  *       difference between the position of the current occurrence in the document and
298  *       the previous occurrence (or zero, if this is the first occurrence in this
299  *       document). If payloads are enabled for the term's field, then PositionDelta/2
300  *       is the difference between the current and the previous position. If payloads
301  *       are enabled and PositionDelta is odd, then PayloadLength is stored, indicating
302  *       the length of the payload at the current term position.</li>
303  *   <li>For example, the TermPositions for a term which occurs as the fourth term in
304  *       one document, and as the fifth and ninth term in a subsequent document, would
305  *       be the following sequence of VInts (payloads disabled):
306  *       <p>4, 5, 4</p></li>
307  *   <li>PayloadData is metadata associated with the current term position. If
308  *       PayloadLength is stored at the current position, then it indicates the length
309  *       of this payload. If PayloadLength is not stored, then this payload has the same
310  *       length as the payload at the previous position.</li>
311  *   <li>OffsetDelta/2 is the difference between this position's startOffset from the
312  *       previous occurrence (or zero, if this is the first occurrence in this document).
313  *       If OffsetDelta is odd, then the length (endOffset-startOffset) differs from the
314  *       previous occurrence and an OffsetLength follows. Offset data is only written for
315  *       {@link IndexOptions#DOCS_AND_FREQS_AND_POSITIONS_AND_OFFSETS}.</li>
316  * </ul>
317  * </dd>
318  * </dl>
319  *
320  * <a name="Payloads"></a>
321  * <dl>
322  * <dd>
323  * <b>Payloads and Offsets</b>
324  * <p>The .pay file will store payloads and offsets associated with certain term-document positions. 
325  *    Some payloads and offsets will be separated out into .pos file, for performance reasons.
326  * <ul>
327  *   <li>PayFile(.pay): --&gt; Header, &lt;TermPayloads, TermOffsets?&gt; <sup>TermCount</sup>, Footer</li>
328  *   <li>Header --&gt; {@link CodecUtil#writeIndexHeader IndexHeader}</li>
329  *   <li>TermPayloads --&gt; &lt;PackedPayLengthBlock, SumPayLength, PayData&gt; <sup>PackedPayBlockNum</sup>
330  *   <li>TermOffsets --&gt; &lt;PackedOffsetStartDeltaBlock, PackedOffsetLengthBlock&gt; <sup>PackedPayBlockNum</sup>
331  *   <li>PackedPayLengthBlock, PackedOffsetStartDeltaBlock, PackedOffsetLengthBlock --&gt; {@link PackedInts PackedInts}</li>
332  *   <li>SumPayLength --&gt; {@link DataOutput#writeVInt VInt}</li>
333  *   <li>PayData --&gt; {@link DataOutput#writeByte byte}<sup>SumPayLength</sup></li>
334  *   <li>Footer --&gt; {@link CodecUtil#writeFooter CodecFooter}</li>
335  * </ul>
336  * <p>Notes:
337  * <ul>
338  *   <li>The order of TermPayloads/TermOffsets will be the same as TermPositions, note that part of 
339  *       payload/offsets are stored in .pos.</li>
340  *   <li>The procedure how PackedPayLengthBlock and PackedOffsetLengthBlock are generated is the 
341  *       same as PackedFreqBlock in chapter <a href="#Frequencies">Frequencies and Skip Data</a>. 
342  *       While PackedStartDeltaBlock follows a same procedure as PackedDocDeltaBlock.</li>
343  *   <li>PackedPayBlockNum is always equal to PackedPosBlockNum, for the same term. It is also synonym 
344  *       for PackedOffsetBlockNum.</li>
345  *   <li>SumPayLength is the total length of payloads written within one block, should be the sum
346  *       of PayLengths in one packed block.</li>
347  *   <li>PayLength in PackedPayLengthBlock is the length of each payload associated with the current 
348  *       position.</li>
349  * </ul>
350  * </dd>
351  * </dl>
352  *
353  * @lucene.experimental
354  */
355 
356 public final class Lucene50PostingsFormat extends PostingsFormat {
357   /**
358    * Filename extension for document number, frequencies, and skip data.
359    * See chapter: <a href="#Frequencies">Frequencies and Skip Data</a>
360    */
361   public static final String DOC_EXTENSION = "doc";
362 
363   /**
364    * Filename extension for positions. 
365    * See chapter: <a href="#Positions">Positions</a>
366    */
367   public static final String POS_EXTENSION = "pos";
368 
369   /**
370    * Filename extension for payloads and offsets.
371    * See chapter: <a href="#Payloads">Payloads and Offsets</a>
372    */
373   public static final String PAY_EXTENSION = "pay";
374   
375   /** 
376    * Expert: The maximum number of skip levels. Smaller values result in 
377    * slightly smaller indexes, but slower skipping in big posting lists.
378    */
379   static final int MAX_SKIP_LEVELS = 10;
380 
381   final static String TERMS_CODEC = "Lucene50PostingsWriterTerms";
382   final static String DOC_CODEC = "Lucene50PostingsWriterDoc";
383   final static String POS_CODEC = "Lucene50PostingsWriterPos";
384   final static String PAY_CODEC = "Lucene50PostingsWriterPay";
385 
386   // Increment version to change it
387   final static int VERSION_START = 0;
388   final static int VERSION_CURRENT = VERSION_START;
389 
390   private final int minTermBlockSize;
391   private final int maxTermBlockSize;
392 
393   /**
394    * Fixed packed block size, number of integers encoded in 
395    * a single packed block.
396    */
397   // NOTE: must be multiple of 64 because of PackedInts long-aligned encoding/decoding
398   public final static int BLOCK_SIZE = 128;
399 
400   /** Creates {@code Lucene50PostingsFormat} with default
401    *  settings. */
402   public Lucene50PostingsFormat() {
403     this(BlockTreeTermsWriter.DEFAULT_MIN_BLOCK_SIZE, BlockTreeTermsWriter.DEFAULT_MAX_BLOCK_SIZE);
404   }
405 
406   /** Creates {@code Lucene50PostingsFormat} with custom
407    *  values for {@code minBlockSize} and {@code
408    *  maxBlockSize} passed to block terms dictionary.
409    *  @see BlockTreeTermsWriter#BlockTreeTermsWriter(SegmentWriteState,PostingsWriterBase,int,int) */
410   public Lucene50PostingsFormat(int minTermBlockSize, int maxTermBlockSize) {
411     super("Lucene50");
412     BlockTreeTermsWriter.validateSettings(minTermBlockSize, maxTermBlockSize);
413     this.minTermBlockSize = minTermBlockSize;
414     this.maxTermBlockSize = maxTermBlockSize;
415   }
416 
417   @Override
418   public String toString() {
419     return getName() + "(blocksize=" + BLOCK_SIZE + ")";
420   }
421 
422   @Override
423   public FieldsConsumer fieldsConsumer(SegmentWriteState state) throws IOException {
424     PostingsWriterBase postingsWriter = new Lucene50PostingsWriter(state);
425 
426     boolean success = false;
427     try {
428       FieldsConsumer ret = new BlockTreeTermsWriter(state, 
429                                                     postingsWriter,
430                                                     minTermBlockSize, 
431                                                     maxTermBlockSize);
432       success = true;
433       return ret;
434     } finally {
435       if (!success) {
436         IOUtils.closeWhileHandlingException(postingsWriter);
437       }
438     }
439   }
440 
441   @Override
442   public FieldsProducer fieldsProducer(SegmentReadState state) throws IOException {
443     PostingsReaderBase postingsReader = new Lucene50PostingsReader(state);
444     boolean success = false;
445     try {
446       FieldsProducer ret = new BlockTreeTermsReader(postingsReader, state);
447       success = true;
448       return ret;
449     } finally {
450       if (!success) {
451         IOUtils.closeWhileHandlingException(postingsReader);
452       }
453     }
454   }

457     long docStartFP = 0;

459     long payStartFP = 0;
460     long skipOffset = -1;
461     long lastPosBlockOffset = -1;
462     // docid when there is a single pulsed posting, otherwise -1
463     // freq is always implicitly totalTermFreq in this case.
464     int singletonDocID = -1;

471     }

479       payStartFP = other.payStartFP;

481       skipOffset = other.skipOffset;

483     }

488     }
